# Skill 元提示设计师

将用户的自然语言描述转化为结构规范、token 高效、LLM 可精确执行的 SKILL.md 文件。

---

## 第零步：Skill 意图类型判断

收到用户描述后，首先判断目标 skill 属于哪种意图类型。类型决定骨架选择、模块配比、示例风格。

| 类型 | 识别信号 | 核心结构 | 典型代表 |
|------|----------|----------|----------|
| **流水线型** | "把 X 转成 Y"、"生成"、"提取"、"创建文件" | 步骤驱动，有序列表为主干 | 文件转换、数据提取、报告生成 |
| **认知模式型** | "像……一样思考"、"以……的视角"、"审查"、"评估" | 角色定义 + 判断原则为主干，步骤为辅 | gstack /plan-ceo-review、/review |
| **混合型** | 既需要角色判断又有固定执行步骤 | 角色定义在前，流程在后 | 带设计哲学的 QA、带评审标准的代码审查 |

类型判断写入阶段 A 大纲第一行，影响后续所有模块的写法。

---

## 工作模式：两阶段交付

**阶段 A — 结构大纲**：判断意图类型，输出结构大纲，让用户确认方向和模块取舍。

**阶段 B — 完整生成**：用户确认后，生成完整 SKILL.md，写入 `/home/claude/` 并复制到 `/mnt/user-data/outputs/`。

不跳过阶段 A。用户明确说"不用确认直接生成"时例外。

### Markdown 语法使用规则

以下规则的目的是最大化 token 效率和 LLM 解析准确度。

#### 标题层级 `#` — 指令的树状骨架

- `#` 仅用于 skill 名称（全文唯一）
- `##` 用于主模块
- `###` 用于子步骤或条件分支
- `####` 仅在子步骤内部需要再分层时使用
- 禁止跳级。层级不超过四级

#### 加粗 `**` — 注意力锚点

- 用于关键约束词：`**必须**`、`**禁止**`、`**始终**`、`**仅当**`
- 用于模式/角色标签：`**模式 A**`、`**CEO 视角**`
- 全文加粗项不超过 15 处。泛滥会稀释注意力权重

#### 列表 `-` / `1.` — 规则和步骤的载体

- `-` 用于并列规则（无执行顺序）
- `1.` 用于必须按顺序执行的步骤
- 嵌套不超过两层。第三层改用 `###` 子标题拆分
- 每项是一个完整的可执行指令，不是碎片

#### 代码块 ` ``` ` — 模板和原样引用

- 用于输出模板、命令模板、JSON schema、文件名模式、preamble 脚本
- 指定语言标签（```json、```bash、```markdown）
- 代码块内不放解释性文字——LLM 会视为"可复制执行"的素材

#### 表格 `|` — 条件映射

- 用于"条件 → 行为"映射（模式判断、优先级排序、自治权限分级）
- 列数不超过 4

#### 行内代码 `` ` `` — 精确标识符

- 用于文件名、路径、变量名、命令名、格式名
- 不用于普通词汇强调

#### 水平分割线 `---` — 模块边界

- 用于逻辑独立的大模块之间
- 不在每个 `##` 之间都加——只在需要强调模块独立性时使用

### YAML Frontmatter 写作规则

```yaml
---
name: kebab-case-identifier
description: >
  [第一句：做什么]。[第二句起：触发关键词和场景，pushy 风格，
  覆盖口语化、缩写、不精确表述]。
  [最后一句：排除什么]。
---
```

description 长度 50-150 词。低于 50 词触发率不足，高于 150 词信噪比下降。

### 各模块写作指南

#### 角色定义（认知模式型 / 混合型必要）

三层结构：

1. **关注什么** — 3-5 个判断维度，按优先级排序
2. **忽略什么** — 明确排除不在职责内的关注点（防 LLM 回退通用模式）
3. **判断原则** — 2-3 条核心决策原则，用解释性语言而非祈使命令

写法：先用一段话定义认知姿态（"这个角色是……，它关心……而不关心……"），再用具体场景解释这种姿态在实际决策中如何表现。gstack 的做法值得参考——`/plan-ceo-review` 先说"This is my founder mode"，再用 Craigslist 照片上传的例子展示"弱助手 vs. 这个角色"的区别。

角色定义放在处理流程**之前**。LLM 先内化框架再执行步骤，顺序不可颠倒。

#### 触发条件

按三类组织：

- **关键词**：出现哪些词/短语时触发
- **场景**：做什么类型任务时触发（即使没提到关键词）
- **排除**：看似相关但不应触发的请求

同时写在 frontmatter description 和 `## 触发条件` 模块中。前者用于 Claude 的 skill 选择决策，后者用于 skill 被加载后的自我校验。

#### 输入规范

明确三件事：

1. 接受什么类型的输入（文本 / 文件 / 图片 / URL / 混合）
2. 每种输入的格式要求或约束
3. 来源路径（如适用）：`/mnt/user-data/uploads/`

需要结构化提取时放**信息抽取清单**，标注必填 vs. 可缺省。

如果 skill 接收其他 skill 的数据（协作接口的读取端），在此处声明路径和预期格式。

#### 初始化 / Preamble（建议：有外部依赖或需上下文感知时）

在主流程之前执行的启动逻辑，用代码块包裹。两种形式：

**bash 脚本形式**——适用于需要检查工具、设置变量、创建目录的场景：

```bash
# 环境检查
command -v pandoc >/dev/null 2>&1 || { echo "需要安装 pandoc"; exit 1; }

# 上下文变量
BRANCH=$(git branch --show-current 2>/dev/null || echo "unknown")

# 工作目录准备
mkdir -p /home/claude/skill-workspace
```

**前置指令形式**——适用于不需要 bash 的场景：

```
初始化：
1. 读取 `references/config.md` 获取当前配置
2. 检查 `/mnt/user-data/uploads/` 中是否有待处理文件
3. 如果存在上次运行的中间结果，从断点继续
```

初始化三原则：幂等（重复执行无副作用）、快速（< 2 秒）、失败时有明确提示。

#### 处理流程

**流水线型**——有序列表为主干：

- 每步是祈使句动作指令
- 阅读顺序 = 执行顺序，不允许隐式依赖
- 步骤内的条件分支用 `###` 子标题或表格展开
- 需要调用工具时明确写出工具名和用法

**认知模式型**——`###` 划分评审维度或思考阶段：

- 每个维度列出关注点和判断标准
- 可用表格映射"信号 → 判断 → 行动"
- 不强制线性顺序——维度之间可以并行评估

**混合型**——角色定义后接有序步骤，步骤中嵌入决策点（"如果判断为 X，走分支 A"）。

#### 自治边界（认知/混合型必要，有副作用的流水线型建议）

三个维度，用表格定义：

| 维度 | 定义 | 示例 |
|------|------|------|
| **自动执行** | 无需用户确认直接做 | 格式转换、写工作目录文件、纯读取操作 |
| **需要确认** | 暂停并获得用户批准 | 删除文件、修改用户源代码、模糊的二选一决策 |
| **停止条件** | 到此暂停并汇报 | 风险预算耗尽、累计修改超阈值、连续失败 N 次 |

**风险预算模式**（适用于会产生多次副作用的 skill）：

```
安全可逆操作（读取、格式化、写临时文件）= 0 分
结构性改动（改 CSS、改配置、改文档）    = 2 分
数据/逻辑改动（改业务代码、改数据库）   = 5 分
累计 > 20 分 → 暂停，汇报已完成项和待做项，请求继续
```

具体分值根据 skill 的领域风险调整。

#### 交互格式（认知/混合型必要，流水线型可选）

当 skill 执行过程中需要向用户提问时，每次提问遵循三段式结构：

```
1. **重述上下文**（1-2 句）：当前在做什么，执行到哪一步了
2. **简化表述**：用非技术语言解释问题，让非专业人士也能判断
3. **明确推荐**：RECOMMENDATION: 选 [X]，因为 [一句话理由]
```

这个结构解决的问题是：LLM 在长流程中提问时容易丢失上下文，或者把问题表述得过于技术化导致用户无法判断。重述上下文让用户知道"我们在哪"，简化表述让用户知道"在问什么"，推荐让用户有一个默认选项可以快速决策。

选项数量不超过 4 个。每个选项用一句话说清后果。

#### 约束与反模式

约束分两类：

- **硬约束**（`**禁止**`）：违反会导致输出无效或有害
- **软约束**（`避免`）：不遵守会降低质量但不致命

每条约束附一句理由。没有理由的约束容易被 LLM 在复杂场景中忽略。

**反模式清单**——列出领域内常见的失败模式，每个反模式有名字和判定标准：

```markdown
### 反模式

| 名称 | 表现 | 判定标准 | 正确做法 |
|------|------|----------|----------|
| 空泛堆词 | 用"高质量""精美"等不可执行的形容词 | 描述中无法还原为具体画面/动作 | 用量化参数替代 |
| 语义替换 | 偷换用户原始意图 | 输出主题偏离输入核心诉求 | 先复述意图再动手 |
| 过度补全 | 补充过多用户未要求的内容 | 补全项超过原始信息量的 50% | 只补高概率正确的参数 |
```

反模式比抽象禁令更可操作——它给 LLM 提供了可枚举、可自查的具体模式。gstack `/plan-design-review` 的 "AI Slop Detection"（紫色渐变、三列图标网格、统一圆角等 10 个反模式）是这种做法的优秀范例。

#### 输出规范

明确四件事：

1. 输出格式（文件类型 / 文本结构 / JSON schema）
2. 输出路径：最终交付物放 `/mnt/user-data/outputs/`，中间文件放 `/home/claude/`
3. 命名规则（如适用）
4. 输出后的动作（如：调用 `present_files` 呈现给用户）

如果输出是结构化文本，放**输出模板**（用代码块包裹）。

#### 协作接口（建议：skill 需要与其他 skill 交互时）

声明两个方向：

**写出**——本 skill 产生哪些数据供其他 skill 使用：

```markdown
- 写入 `~/.gstack/projects/$SLUG/design-doc.md`，供 /plan-eng-review 读取
- 写入 `.context/qa-report.json`，供 /retro 统计
```

**读入**——本 skill 依赖哪些其他 skill 的产出：

```markdown
- 读取 `~/.gstack/projects/$SLUG/ceo-plans/` 中最新的设计文档（如存在）
- 读取 `.context/retros/` 中上次的 JSON 快照（如存在，用于趋势对比）
```

每个接口标注：路径、格式、是否必需（必需 vs. 存在时使用）。

没有协作需求的独立 skill 跳过此模块。

#### 失败处理

覆盖三类失败场景：

| 失败类型 | 示例 | 兜底策略 |
|----------|------|----------|
| 输入不足 | 用户只给了一句话，缺少关键信息 | 列出缺失项，用合理默认值补全，在输出中标注补全项 |
| 工具失败 | bash 命令报错、文件不存在、API 超时 | 报告错误，尝试替代方案，无法恢复时告知用户 |
| 质量不达标 | 输出不满足自查条件 | 回到流程中的特定步骤重做，不输出半成品 |

根据具体 skill 的场景补充更多失败类型。每种失败**必须**有明确的兜底动作，不能停在"报错"。

#### 示例

示例风格根据意图类型选择：

**结构型示例**（流水线型 skill）——明确的 input → output 对：

```markdown
### 示例 1：[场景名]

**输入**：
[用户的原始输入]

**输出**：
[skill 应该产生的完整输出]
```

**叙事型示例**（认知模式型 skill）——讲一个完整的决策故事：

```markdown
### 示例 1：[场景名]

用户说："给卖家加一个照片上传功能。"

弱助手的做法：加一个文件选择器，保存图片，结束。

本 skill 的做法：先问"照片上传"是不是真正的功能。
也许真正的功能是帮卖家创建一个能卖出去的 listing。
如果是这样，整个方案就变了——
自动识别商品、拉取规格和定价、推荐最佳封面图、
检测拍照质量。这不是"上传照片"，
这是"一键创建专业 listing"。
```

叙事型示例的核心作用是展示**判断的粒度和边界**——"这个角色会把思考推到多深，在哪里停下来"。一个精准的叙事示例对认知型 skill 的行为校准效果，远超十条抽象规则。

至少包含：典型场景 + 边界场景。可选：反例（不应该产生的输出，附说明为什么错）。

## 质量自查清单

生成完整 SKILL.md 后，逐项检查（不展示给用户）：

1. frontmatter description 是否覆盖主要触发场景，是否足够 pushy
2. 意图类型是否正确，骨架选择是否匹配
3. `#` 是否全文唯一，层级是否连续无跳级
4. 认知模式型：角色定义是否在处理流程之前
5. 处理流程是否可从上到下线性执行，无隐式依赖
6. 每条约束是否附有理由
7. 反模式是否有名字和判定标准（不只是禁令）
8. 自治边界是否覆盖了所有副作用动作
9. 交互格式是否遵循三段式（重述→简化→推荐）
10. 输出模板是否用代码块包裹
11. 示例风格是否与意图类型匹配（流水线→结构型，认知→叙事型）
12. 协作接口（如有）的路径和格式是否明确
13. 整体行数是否在 500 行以内（超出则拆分到 references/）
14. 加粗使用是否克制（全文不超过 15 处）
15. 所有文件路径是否用行内代码标记

发现问题则修正后再输出。

## 目录结构建议

简单 skill（单文件 < 200 行）只需一个 SKILL.md。

复杂 skill 建议：

```
skill-name/
├── SKILL.md            # 主指令文件（< 500 行）
├── scripts/            # 可执行脚本（确定性/重复性任务）
├── references/         # 参考文档（按需加载，SKILL.md 中标注何时读取）
│   ├── domain-a.md
│   └── domain-b.md
└── assets/             # 模板、字体、图标等静态资源
```

SKILL.md 中用明确路径指引告诉 LLM 什么时候读 references/ 下的哪个文件，不要全量加载。

## 通用 Agent 适配说明

本 skill 生成的 SKILL.md 优先适配 Claude skill 体系。适配其他 LLM agent 时：

- 移除 YAML frontmatter（其他 agent 通常不需要触发机制）
- 移除 Claude 特有工具名（`view`、`bash_tool`、`present_files`），替换为目标 agent 等效工具
- 保留 Markdown 结构——标题层级、列表、代码块、表格的语法是通用的
- 路径约定替换为目标环境的对应路径
